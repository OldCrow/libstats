## ARM optimized-routines

**Project**: https://github.com/ARM-software/optimized-routines
**License**: MIT OR Apache-2.0 WITH LLVM-exception (used here under MIT)
**Usage**: The production NEON table-based `exp` (Issue #33 Q1) and the
experimental AVX-512 table-based `exp` (Issue #33 Stage 3, dev-tool only) reproduce
the N=128 lookup table and the argument-reduction constants and minimax polynomial
coefficients from ARM optimized-routines `math/exp_data.c` + `math/exp.c`. The
experimental NEON table-based `log` (Issue #33 Q1, dev-tool only -- not
productionized; see PLAN.md) similarly reproduces `math/log_data.c` + `math/log.c`.

Affected files:
- `scripts/gen_neon_exp_table.py` ŌĆö regenerates the **production** NEON exp table
  (same values as ARM's table, re-interleaved to an Array-of-Structs layout) and
  cross-checks it bit-exactly against ARM's published values
- `src/neon_exp_data.inc` ŌĆö the generated **production** NEON exp table
  (`kExpNeonTable`), included by `vector_exp_neon` in `src/simd_neon.cpp`
- `scripts/gen_avx512_exp_table.py` ŌĆö regenerates the AVX-512 exp table and
  cross-checks it bit-exactly against ARM's published values (dev tool only)
- `src/avx512_exp_data.inc` ŌĆö the generated AVX-512 exp table (`kExpSbitsAvx512`,
  `kExpTailAvx512`; dev tool only)
- `tools/gather_throughput_probe.cpp` ŌĆö `vectorExpAvx512Gather` (the 8-wide
  gather port; opt-in dev tool only, not part of the shipped library)
- `scripts/gen_neon_log_table.py` ŌĆö embeds ARM's `__log_data.tab` (N=128)
  verbatim and cross-checks every entry against an independently computed
  `log(1/invc)` reference (dev tool only -- the ARM-port log was not
  productionized)
- `src/neon_log_data.inc` ŌĆö the generated NEON log table (`kLogNeonTable`; dev
  tool only)

- `tools/neon_table_transcendental_probe.cpp` ŌĆö `vectorExpNeonGather` (the
  prototype that became the production kernel above) and `vectorLogNeonGather`
  (still opt-in dev tool only, not part of the shipped library)

Note (2026-07-19): the **production** NEON `vector_log_neon` was subsequently
replaced with an independently derived clean-room kernel
(`src/neon_log_cleanroom_data.inc`, `scripts/gen_neon_log_cleanroom_table.py`)
that carries **no third-party source** and supersedes the Q1 "performance
null" verdict for log (better on both accuracy and speed axes); see
`docs/NEON_LOG_DERIVATION.md` and `docs/NEON_LOG_DIVERGENCE_AUDIT.md`. The
ARM-derived dev-tool files above are unchanged and remain MIT-attributed.

This deliberately derives from the permissively licensed ARM optimized-routines

## `vector_erf_neon` ŌĆö independently derived (Issue #67)

`vector_erf_neon` (`src/simd_neon.cpp`), its table (`src/neon_erf_data.inc`), and its
generator (`scripts/gen_neon_erf_table.py`) were, through v2.0.4, a port of glibc's
**LGPL-2.1+** `sysdeps/aarch64/fpu/erf_advsimd.c`. On 2026-07-19 they were **replaced**
by an implementation derived independently from first principles ŌĆö a local Taylor
expansion about a uniform grid, using the Hermite-polynomial structure of the
derivatives of `exp(-x^2)` ŌĆö authored clean-room from a functional specification with no
access to any existing erf implementation. The replacement carries **no third-party
source** (it is original work, and more accurate than what it replaced: max 1 ULP vs
~2.29 ULP); it is documented here only to record the resolution, not as a dependency.

Provenance record: `docs/NEON_ERF_DERIVATION.md` (the derivation) and
`docs/NEON_ERF_DIVERGENCE_AUDIT.md` (a point-by-point comparison confirming no shared
*expression* with the former glibc code ŌĆö only mathematically forced coefficient values
and a generic method coincide). This is an engineering/process record, not a legal
opinion.

## Numerical Recipes (algorithmic reference only)

**Reference**: Press, W.H. et al., *Numerical Recipes in C++*, Cambridge
University Press, 3rd edition 2007.

**Usage**: The following functions in `src/math_utils.cpp` use the same
standard numerical methods described in Numerical Recipes:
- `erf_inv` ŌĆö rational approximation for the inverse error function
  (Moro/Acklam method, also described in NR)
- `beta_continued_fraction` ŌĆö Lentz modified continued-fraction evaluation
  of the regularized incomplete beta function
- `gamma_p_series` ŌĆö power-series expansion of the regularized incomplete
  gamma function

These algorithms are classical results (Lentz 1976; series predating NR)
and are described in many references. The implementations in this library
are original code written independently; no code was copied from the NR
publication. NR is listed here as an acknowledgment of the algorithmic
references consulted during development.
